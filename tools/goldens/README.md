# MF6.1 golden generation

Golden CSVs are the numerical oracle for the MF6.1 force model: the Rust kernels
(`crates/outlap-tire`) and the Python forward model must reproduce them to within the tolerance
below (HANDOFF §12/§13, the `Fx/Fy/Mz ≤ 0.5%` gate).

## Clean-room policy (read before touching this)

The goldens are generated by an external Magic-Formula implementation whose **outputs are used as
data only**. We never read, port, or vendor that implementation's source — outlap's MF6.1 is
derived from the Pacejka book alone (§15). The generator (`gen_mf61_goldens.m`) is our own
authorship (AGPL); it `addpath`s a **user-supplied** oracle checkout that is never committed here.

## Oracle

The committed goldens were generated with **teasit/magic-formula-tyre-library** (GPL-3.0), the
`magicformula.v61` package, run under **GNU Octave**. GPL is fine here: the library is executed as
a tool and only its numeric outputs are captured (never its source). We call `magicformula.v61.eval`
with a parameter struct directly, bypassing the object/`.tir` reader. The exact oracle commit and
Octave version are recorded in each CSV's provenance header (asserted by a test).

If teasit is unavailable, MFeval (Marco Furlan) under Octave is an equivalent oracle (read its
LICENSE first); digitised Pacejka book figures are the last-resort fallback at a looser,
separately-labelled gate.

## CSV contract

One file per channel-group per tyre under `crates/outlap-tire/tests/golden/<tyre-slug>/`:
`fx0.csv`, `fy0_mz.csv`, `combined.csv`. One row per evaluation point, inputs left, outputs right,
**SI units, ISO 8855 sign convention**. Every file starts with `#` provenance header lines; the
oracle must be pinned to a **commit** (`oracle: … @ <hash>`), enforced by a test:

```
# generator: tools/goldens/gen_mf61_goldens.m, oracle: <name> @ <commit> (<license>), GNU Octave <version>
# tyre: <slug> (Pacejka 2006 Table A3.1), ISO 8855 sign
kappa,alpha_rad,gamma_rad,fz_n,p_pa,vx_mps,fx_n,fy_n,mz_nm,mx_nm,my_nm
```

## Sweep grid (kept ≲ 1 MB/tyre)

- `fx0.csv`: κ ∈ 41 pts [−0.30, 0.30] × Fz ∈ {0.5, 1, 1.5, 2}·FNOMIN × γ = 0, α = 0.
- `fy0_mz.csv`: α ∈ 41 pts [−0.21, 0.21] rad (≈ ±12°) × same Fz set × γ ∈ {−4°, 0, 4°}, κ = 0.
- `combined.csv`: κ ∈ 11 × α ∈ 11 at Fz = FNOMIN, γ = 0.
- `V = LONGVL` throughout. Pressure is held at NOMPRES (the 2nd-edition reference tyre carries no
  `PP*` terms, so a pressure sweep exercises nothing — see its README).

## Tolerance (in the test code, not the CSV)

Per point, `|model − ref| ≤ max(0.005·|ref|, floor)`, with a per-channel absolute floor so the
zero crossings (Mz, and Fx/Fy through zero slip) don't divide by ~0:

- Fx: `floor = 0.005 · PDX1 · Fz`
- Fy: `floor = 0.005 · |PDY1| · Fz`
- Mz: `floor = 0.005 · max_row|Mz_ref|` within the Fz bin

The Rust golden test (`crates/outlap-tire/tests/golden.rs`) uses this rule; the Python cross-check
will reuse it verbatim when the Python forward model lands (M2 PR7).

## Regenerating (the repo's golden-file governance)

These are external-oracle data, so there is no in-tree `--bless`. To regenerate:

1. `MF_ORACLE_SRC=/path/to/oracle/src ./run.sh` (stages the package, runs Octave, rewrites the CSVs).
2. Re-commit the CSVs **only in a PR that updates the provenance headers and states the
   physics/tooling reason** for the change. A CSV diff with no such note is a review stop.

## Status

The goldens for `pacejka_2006_205_60r15` are committed and gated at ≤ 0.5% in
`crates/outlap-tire/tests/golden.rs`. The reference tyre additionally has the load + physical-sanity
gate in `crates/outlap-tire/tests/reference.rs`.
